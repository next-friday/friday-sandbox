# Component blueprint

The design station for `@friday-sandbox/react` base components: choose the primitive and author its **transient `ComponentSpec`** — the primitive, the variants, the tokens, the layout, the parts, the demo tree, the assets, and the prose — against the repo's existing vocabulary, then ship it in a **GitHub issue** so `component-implement` can generate the component from it. A precise spec is the leverage; a vague one produces a vague component. Grill one decision at a time; the spec is done when every field `ComponentSpec` requires has a grounded value, nothing left to guess at generation time. Recommend only what the docs, the live tree, or the generator's own types ground — the primitive's real props, the existing component it is most like. Never recommend, pre-fill, label "recommended", or float example values for what the human exposes — the prop, variant, size, and state set is the human's spec; elicit it with an open prompt and record it verbatim into the spec's fields. A suggested value anchors the human and corrupts the spec exactly as a guess would.

```text
design interview → transient ComponentSpec (primitive + variants + tokens + layout + parts + demo + assets + prose)
  └→ shipped as a GitHub issue — the spec's source embedded verbatim, plus a short summary
       └→ component-implement receives #issue: branch → run the generator on the spec → fill the residual → gates → PR (Closes #issue)
```

## When to use

- Before `component-implement`, for a new or reworked component whose primitive, variant model, or token mapping is not decided, or whose issue/spec does not exist yet.
- Symptoms: unsure which primitive to wrap, how many roles/sizes, which states, or what the demo composition looks like.
- Not this station:
  - Building or shipping the component (run the generator, branch, gates, PR) → `component-implement`.
  - Bot-review comments after the PR opens → `component-rebut`.

## Steps

Steps 1–6 settle the spec (no GitHub write); steps 7–8 buffer and ship it, gated on explicit authorization.

1. **Premise — the spec comes from the human; never guess it.** Don't presuppose the component or pre-fill any decision from your own context. Ask the user four things, in order, and treat each answer as the authoritative spec: (a) **which component** (name); (b) **which primitive** it wraps — react-aria-components, radix-ui, native HTML, or an existing `bases/` component it composes (step 2); (c) **which props and variants to expose** — the prop set, the color/variant/size/state ladder, the boolean modifiers; (d) **any extra requirements** — an open free-text prompt for anything (a)–(c) don't capture: specific behaviors, accessibility needs, constraints, edge cases. (d) is optional but always offer it; fold whatever they give into the spec's fields, never invent extra requirements they didn't state. Never invent the exposed prop or variant set, and never derive it from the library docs alone: the library docs (step 2) **ground** the answers — they confirm the part's real props and types — but they do **not** replace the ask. A prop or variant the human leaves unspecified is a question to ask, never a gap to fill yourself — and ask it open: no menu of candidate values, no "recommended" default. Then settle repo-fit with the user: whether it belongs here or extends a `bases/` component instead of duplicating (menu: `button`, `text`, `flex`/`grid`, `scroll-area`, `separator`, `spinner`, `surface`, `link`). Confirm repo-fit against the live tree, not memory — list `packages/react/src/components/bases/` for what exists. Design only from that live tree, the human's spec, `packages/component-generator/src/component-spec.ts` (the `ComponentSpec` type every field below fills), and the library docs; never seed the shape from git history or a prior/dropped version — stale shape corrupts a fresh design.
   - **Then design it twice before converging** — if the shape isn't obvious, diverge on the API and visual structure (see [`references/DESIGN-IT-TWICE.md`](references/DESIGN-IT-TWICE.md)), on parallel **opus** sub-agents (design divergence is judgment), pick or hybridize a winner, and carry it into the fields below. Skip when grilling already settled the shape in one pass.

2. **Pick `primitive.kind`, then the interactivity flags — two separate decisions.** The sanctioned set is **closed to what `packages/component-generator/src/registry.ts` registers today: `native`, `react-aria`, `radix`** — nothing else; a spec naming any other kind is rejected at validation (`validate.ts`'s `checkKindRegistered`). Ground the chosen part in that library's own docs, not memory — each publishes an LLM-readable markdown page per component; **fetch it** to confirm the real props, render-prop values, and data attributes before you map them:
   - `react-aria` → `https://react-aria.adobe.com/<Part>.md` (index: `https://react-aria.adobe.com/llms.txt`) — the part's name becomes `element: { wraps: "<Part>" }` on the root or a subpart; the emitter imports it from its own deep subpath `react-aria-components/<Part>`.
   - `radix` → `https://www.radix-ui.com/primitives/docs/components/<name>.md` — set `primitive.part` to the radix namespace (e.g. `"Avatar"`) and each part's `element: { wraps: "<Sub>" }` (e.g. `"Root"`, `"Image"`, `"Fallback"`); the emitter imports `{ <part> as Radix<part> } from "radix-ui"` and renders `Radix<part>.<wraps>`.
   - `native` (no library — `kbd`, `code`, `mark`, `samp`, or a react-aria part with no interaction state such as `Label`/`Text`) → `element: { native: "<tag>" }`, grounded against the element's MDN page.

   When several candidate primitives are in play, fan out one fetch sub-agent per candidate in a **single message** rather than fetching serially; each returns the part's real props, render-prop values, and data attributes, no edits, for the main thread to weigh. Size sub-agents to the task throughout this skill: **sonnet** for mechanical fetch-and-read like this, **opus** for judgment (the design-twice divergence and the plan review), **haiku** for a lone grep.

   Not every part has its own doc page: check the library's `llms.txt` index for the real path first. A missing doc page never means a missing type — resolve the installed `.d.ts` directly (from `packages/react`, run `node -e "console.log(require.resolve('react-aria-components/package.json'))"`, then read `dist/types/src/<Part>.d.ts` beside it). Set `primitive.interactive: true` only when the part carries focus/hover/press/keyboard state — it drives the ramp-based size axis and the `composeTailwindRenderProps` wiring in what gets emitted; set `primitive.client: true` whenever a react-aria or radix part is wrapped (the emitter adds `"use client"`). Interactivity is a flag orthogonal to which library is wrapped, not a proxy for it: a **non-interactive** react-aria part (`Label`, `Text`) is still `kind: "react-aria"` with `interactive: false`.

   **A composed `bases/` component stands in for the primitive.** When step 1(b) chose to compose an existing base instead of wrapping a headless part, that base is the primitive: no library fetch — ground against the base's own props and variant ladder (`<base>.tsx` / `<base>.styles.ts`), record `primitive.kind: "native"`, and record only which props the composition fixes, forwards, or adds.

3. **Author `variants` per part.** First decide whether the component is colored/interactive or structural. A **colored/interactive** component (button-like) reuses the full vocabulary: the shared `roleColor` axis (`primary secondary accent info success warning danger`, `mechanism: "role"`), a `variant` axis (`solid subtle surface outline ghost plain`), a `size` axis, boolean modifiers. A **structural/typography** component (label, text, separator-like) **drops the color and visual-variant axes entirely** — state that as the reason — and uses only its domain axis (`typeScale`, an orientation) plus boolean state modifiers. Import a repeated axis from `packages/component-generator/src/axes.ts` (`roleColor`, `typeScale`, `size(upTo)`) rather than re-listing its values inline — the catalog is the single source for the repo's color/typography/size vocabulary; add a bespoke axis only with a stated reason, as a plain `Axis` object (`values`, `default`, `boolean?`, `cascade?`, `mechanism?`, `prop?`), every value distinct. Never author a class name yourself — the emitter derives every value's class as `fri-<name>[-<part>]-<value>`. Give a non-plain axis its `mechanism` (`role` | `paint` | `radius-scale` | `utility` | `stacking`) so the emitter knows how to expand it in CSS; `stacking` is exactly how a group-overlap behavior (Avatar's stacking) becomes a spec-declared mechanism instead of a hand-written `Children.map`/`cloneElement` the repo's lint forbids.

4. **Bind `tokens` and `layout`.** `tokens: TokenBinding[]` on a part maps a slot (`background` | `foreground` | `border` | `ring`) to a `--fri-<binds>` token name (`binds`, without the `--fri-` prefix) — reuse the existing ladder (the seeds, the derived hover/pressed/tint steps; the full set is in `button.css`), never invent a color. `layout.size` is a `SizeSource`: `{ kind: "ramp", n: { <step>: <multiplier>, … }, md: <multiplier> }` for a component-local ramp (matching `button`'s `--_button-n`), or `{ kind: "scale" }` to bind directly to the shared `--fri-scale-*` steps (matching a container like `surface`). Set `layout.radiusArchetype` (`action` | `field` | `box`) to drive the shared radius formula — the emitter computes `--<name>-radius: calc(var(--fri-<archetype>-radius) * <ratio>)` per step at generate time (CSS cannot divide length by length, so the ratio is a numeric literal baked in); never author the radius `calc()` or its ratio by hand. `spacing: ("gap" | "padding")[]` on a part opts into the shared spacing utilities (the one mirror-exempt exception) instead of a per-component token. Ground every token binding and every `role`/`radius-scale`-mechanism axis value against the real names now — `packages/component-generator/src/tokens.ts` reads `packages/styles/src/themes/{shared/scales.css,shared/variables.css,default/tokens.css}` and rejects a name absent from those files at generation time.

5. **Declare `parts` and `compound` for a multi-part component.** A single-part component sets neither. A multi-part component sets `parts: Part[]` (each with its own `role`, `element`, and optionally its own `variants`/`tokens`) and `compound`: `"callable-root"` for a callable component whose subparts hang off it by dot-access (`Link.Icon`, `Grid.Item` — the emitter wires the `Object.assign` namespace and both barrel export forms), or `"namespace-of-parts"` for parts with no callable root (`ScrollArea`-shaped — the emitter exports each part under its own name; no hand-wiring either way). `required: string[]` on a part names its non-optional pass-through props; `manualChildren: true` marks a part whose demo content is a literal, not a sample or asset (`Avatar.Fallback`'s initials). `behaviors: Behavior[]` (`{ kind: "overlap" }`, `{ kind: "separator-ring", default: boolean }`) records a fixed composition mechanism the emitter implements — never a hand-sketched one.

6. **Record `assets`, `demo`, and `prose`.** `assets: { url, alt? }[]` holds every human-provided URL or value verbatim — record it exactly as given, never substitute a placeholder service. `demo: { rootProps?, tree: DemoNode[] }` is the low-entropy composition contract for a compound or a container: each `DemoNode` names a `part` (omit for a direct child of the root), `props`, and `content` (`{ asset: n }` to bind `assets[n]`'s `url`/`alt` into that node, `{ sample: "Boxes" | "Lorem" | … , props? }` for a `@friday-sandbox/react/samples` fixture, or `{ text }` for a literal), or nested `children`. Because the emitter renders every node's content directly from the spec, a referenced asset cannot be silently dropped or swapped — record the human's URL and reference it by index; there is no derivation step where it could be replaced. A leaf, text-content component (`<Button>Button</Button>`) needs no `demo` — its showcases derive from `variants` alone. `prose: { purpose, whenToUse: string[], whenNotToUse: string[] }` is the irreducible free text; write it in `.claude/rules/prose-style.md` voice — it lands in the emitted mdx unedited.

7. **Converge, then buffer the spec locally.** Only after the spec has CONVERGED (user approved in chat, no open decisions) does it leave chat: write it as a real `.ts` module conforming to `ComponentSpec` (`packages/component-generator/src/component-spec.ts`) at `~/.claude/projects/<project>/specs/<name>.spec.ts` — the per-project dir where your memory lives, outside the repo, persistent and resumable across sessions, never committed. This file **is** the design; there is no separate prose draft to keep in sync with it. Self-review it against the type — every field present, every `default` a member of its own `values`, `compound` set iff `parts` is non-empty, every primitive kind one of the three registered (see [Verify](#verify)).

8. **Record the spec to the issue, gated on explicit authorization.** Only with an **explicit, per-artifact go-ahead** to write to the shared tracker — until then the spec stays at the path in step 7. Preflight `gh`, ASK before create (`create a new issue titled "<X>"?` or `update #N?`) — never auto-match by title — fill the repo's issue template via `--body-file`, title per the repo's enforced convention. The body carries: the spec's `.ts` source verbatim in a fenced code block (the load-bearing artifact `component-implement` runs the generator against — embed it, never re-describe it in prose, so nothing has to be re-derived later), a short human-readable summary (primitive, variant model, demo shape) for a reader skimming the issue, and a note of the residual `component-implement` still owns by hand: prose polish and the `<name>.play.ts` interaction test. No placeholders: "the usual variants" or "like Button" → the spec already enumerates every value with its token; the issue only restates it for a human reader. Self-review the recorded body: placeholders (TBD/TODO) gone, the embedded spec parses as the type it claims, no internal contradiction. Fix by editing the body (`gh issue edit <n> --body-file`); GitHub keeps the edit history.

## Done — the contract

- [ ] Premise resolved — exists, or extends a `bases/` pattern — with the real problem named.
- [ ] Component name, primitive, the prop/variant spec, and any extra requirements all came from the human's answers, validated against the library docs and `component-spec.ts` — nothing guessed or derived from docs alone.
- [ ] `primitive.kind` is one of the three registered kinds (`native`, `react-aria`, `radix`), justified against the existing component it is most like.
- [ ] Every axis in `variants` reuses the repo vocabulary (or the shared `axes` catalog), every value distinct — its class is the emitter's to derive, never hand-authored.
- [ ] Every `tokens` binding and every role/radius-scale axis value resolves to a real `--fri-*` name (checked against the three token files).
- [ ] `parts`/`compound` set together or neither; `demo` present for every compound or container; every `assets` entry referenced by a `demo` node.
- [ ] Spec **shipped as a GitHub issue** (after explicit per-artifact authorization) with its `.ts` source embedded verbatim, its `#N` handed to `component-implement`. The local file (`~/.claude/projects/<project>/specs/<name>.spec.ts`) is only the pre-ship buffer — a converged spec that stops there is **paused awaiting authorization, not done**. No code or build action taken; running the generator is `component-implement`'s job.

## Verify

- Re-read the spec against `component-spec.ts`: every field present, every `default` a member of its own axis's `values`, `compound` set iff `parts` is non-empty (`validate.ts`'s own checks) — a spec that fails these throws before a single file is written, so catching it here costs nothing.
- Re-read every `tokens` binding and role/radius-scale axis value against `packages/styles/src/themes/{shared/scales.css,shared/variables.css,default/tokens.css}` — a name not in those files is not a real token.
- Re-read `demo` against `assets`: does every asset index resolve to an entry that exists, and does every human-provided asset appear in at least one node?
- Sub-agent reviewer (**opus** — adversarial judgment): pressure-test the primitive choice, the variant model, and the demo tree for gaps — a missing state, an ambiguous axis, an asset never referenced — before any build. Builder ≠ verifier: don't have the spec's author grade it.

## Closing summary table

On finishing, report one markdown table — the result at a glance:

| Issue  | Primitive                                   | Variants                 | Sizes     | Tokens               | States         |
| ------ | ------------------------------------------- | ------------------------ | --------- | -------------------- | -------------- |
| `#<n>` | `native`/`react-aria`/`radix` — `<wrapped>` | `<roles>` × `<variants>` | `<xs…xl>` | `--fri-<role>` slots | `<data-attrs>` |

## Red flags — STOP

- Deciding variants before the primitive → the primitive gates the props; choose it first.
- Naming a `primitive.kind` other than `native`, `react-aria`, or `radix` → the registry rejects it; there is no fourth sanctioned kind today (adding one is registering a new adapter, not a spec decision).
- Inventing a color or size vocabulary when the repo set or the `axes` catalog fits → import `roleColor`/`typeScale`/`size()`, don't re-list values inline.
- Forcing the color or visual-variant ladder onto a structural component (label, text, separator) → drop those axes; use the domain axis plus boolean states.
- Hand-deriving a class name, a radius `calc()`, or a ramp ratio in the spec → those are the emitter's job; record the axis/token/archetype and let generation compute them.
- Guessing the component, primitive, prop, or variant set — or pre-filling it from the library docs alone — instead of getting it from the human → STOP; the spec comes from the human, the docs only ground it.
- Recommending, suggesting, or offering a "recommended" option or example menu for what to expose (props, variants, sizes, states) → STOP. A floated value anchors the human as badly as a guess.
- Authoring a `demo` node that references an asset index with no matching `assets` entry, or recording a human-provided asset with no `demo` node referencing it → STOP; the asset would be silently absent from the emitted output.
- "Figure out variants while coding" → a vague spec propagates into every emitted surface; fix it here.
- Recording to the shared tracker before the spec has converged, or without an explicit per-artifact yes → STOP. A design approval is not a write authorization.
- Matching an issue to the request by title or topic resemblance → STOP; confirm the number with the user.
- Telling anyone to hand-write `.tsx`, `.styles.ts`, `.css`, `.stories.tsx`, or `.mdx` → those are emitted by `component-implement` running the generator on this spec; the spec is the only artifact this station produces.

## What this encodes

- The existing `bases/` components are the pattern menu for repo-fit and for grounding a bespoke axis: `button`, `text`, `flex`, `grid`, `scroll-area`, `separator`, `spinner`, `surface`, `link`.
- `packages/component-generator/src/component-spec.ts` is the `ComponentSpec` type this whole skill fills; `src/axes.ts` is the shared axis catalog; `src/registry.ts` is the closed set of registered primitive kinds; `src/validate.ts` throws on an incoherent spec before any file is written.
- The `--fri-*` tokens are hand-authored in `packages/styles/src/themes/` — per-mode color seeds in `<name>/tokens.css`, theme-agnostic scales in `shared/scales.css`, the derived ladder in `shared/variables.css`.
- The issue body is the single source of truth for the spec (GitHub keeps its edit history); `component-implement` branches from it, runs the generator, and ships the PR that closes it. The tracker is shared (`CONTRIBUTING.md`) — every write needs an explicit per-artifact yes, and never touch an issue this session did not create.
