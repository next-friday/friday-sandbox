# Component blueprint

## Purpose

Design a `@friday-sandbox/react` base component — the primitive it wraps, the variant ladder, the token/state map, the surface/demo plan — and record the approved design plus an implementation plan in a GitHub issue as `component-implement`'s branchable target. Building and shipping the component is `component-implement`; bot-review comments after the PR opens are `component-rebut`.

## Input

- A request to design a base component; the prop, variant, size, state, and token spec comes from the human.

## Tasks

Steps 1–5 settle the design (no GitHub write); steps 6–7 record it to the issue, gated on explicit authorization.

1. **Get the spec from the human.** Ask the user four things, in order, and treat each answer as the authoritative spec: (a) **which component** (name); (b) **which primitive** it wraps — react-aria-components, Radix UI, Base UI, native HTML, or an existing `bases/` component it composes (step 2); (c) **which props and variants to expose** — the prop set, the color/variant/size/state ladder, the boolean modifiers; (d) **any extra requirements** — an open free-text prompt for anything (a)–(c) don't capture (behaviors, accessibility needs, constraints, edge cases). (d) is optional but always offer it; fold whatever they give into the spec. A headless part's own props pass through (`ComponentPropsWithRef<typeof X>`); the design decides which variants, sizes, states, and custom props to add on top. Settle repo-fit with the user: whether it belongs here or extends a `bases/` component instead of duplicating (menu: `button`, `text`, `flex`/`grid`, `scroll-area`, `separator`, `spinner`, `surface`, `link`) — confirm against the live tree by listing `packages/react/src/components/bases/`.
   - **Design it twice before converging** — if the shape isn't obvious, diverge on the API and visual structure ([`references/DESIGN-IT-TWICE.md`](references/DESIGN-IT-TWICE.md)) on parallel **opus** sub-agents, pick or hybridize a winner, and carry it into the steps below. Skip when grilling settled the shape in one pass.
2. **Pick the primitive, then the generator kind — two separate decisions.** First, what `<name>.tsx` wraps. The sanctioned headless sources are **react-aria-components**, **Radix UI**, and **Base UI** (or native HTML for pure layout/text); whichever you pick must be a workspace dependency. Fetch that library's own LLM-readable markdown page to confirm the real props, render-prop values, and data attributes:
   - react-aria-components → `https://react-aria.adobe.com/<Part>.md` (index: `https://react-aria.adobe.com/llms.txt`) — e.g. `button` wraps `AriaButton`.
   - Radix UI → `https://www.radix-ui.com/primitives/docs/components/<name>.md` — e.g. `scroll-area` wraps a Radix part.
   - Base UI → `https://base-ui.com/react/components/<name>.md` (index: `https://base-ui.com/llms.txt`).
   - native HTML (`kbd`, `code`, `mark`, `samp`) → ground against the element's MDN page.

   Several candidate primitives → fan out one fetch sub-agent per candidate in a **single message**; each returns the part's real props, render-prop values, and data attributes, no edits. Size sub-agents: **sonnet** for mechanical fetch-and-read, **opus** for judgment (design-twice divergence, plan review), **haiku** for a lone grep; the author never grades its own work. Not every part has its own page — check the `llms.txt` index for the real path; a sub-part may live inside a parent page, exist only as a hook (`useLabel`), or have no page (react-aria's `Label` has no `Label.md` and isn't in `llms.txt`). A missing doc page still ships an **installed TypeScript type, the authoritative source** — read it directly. Resolve the path (pnpm — the part lives under `node_modules/.pnpm/…`): from `packages/react`, run `node -e "console.log(require.resolve('react-aria-components/package.json'))"`, then read `dist/types/src/<Part>.d.ts` (or `dist/types/exports/<Part>.d.ts`) beside it. A headless part exposes only its own props — required/invalid/disabled are additions you design here.

   Second, the generator's `primitive` prompt is **interactivity, not library**: pick `aria` only for an interactive component (focus/hover/press, a size ramp); pick `native` for any display/layout component, **including a react-aria primitive that carries no interaction state** (Label and Text wrap react-aria parts but scaffold as `native`, then swap the placeholder `<div>` for the part in `component-implement`). Choose the wrapped primitive first. Record the import as the part's **per-component subpath** (`react-aria-components/<Part>`, e.g. `react-aria-components/Label`), not the barrel; `component-implement` adds the matching `optimizeDeps.include` entry in `.storybook/main.ts`.

   **A composed `bases/` component stands in for the primitive.** When (b) chose to compose an existing base, that base is the primitive: no library fetch — ground against the base's own props and variant ladder (`<base>.tsx` / `<base>.styles.ts`). Scaffold `native` and swap the placeholder for the base in `component-implement`; record only which props it fixes, forwards, or adds.

3. **Resolve the variant ladder** — first decide colored/interactive or structural. A **colored/interactive** component (button-like) reuses the full vocabulary: color roles `primary secondary accent info success warning danger`; visual `variant` `solid subtle surface outline ghost plain`; `size` `xs sm md lg xl`; boolean modifiers. A **structural/typography** component (label, text, separator-like) **drops the color and visual-variant axes entirely** — state that as the reason — and uses only its domain axis (a typography scale, an orientation) plus boolean state modifiers; mirror the **axis ladder** of `text.styles.ts` / `separator.styles.ts`, not `button`'s color/variant axes. Record the `tv()` **shape the generator emits** — a flat `tv({ base: "fri-<name>", variants })` — even when borrowing that ladder: `text`/`separator` predate the generator and use `slots: { root }`, but `component-implement` keeps the flat `base` form, so never record `slots`. Add a new axis only with a stated reason, every value distinct, each with its `fri-<name>-<value>` class.
4. **Map tokens and states.** A **colored** component maps each color → the `--fri-<role>` slots (the four seeds `-foreground`, `-secondary`, `-secondary-foreground`, plus the derived `-hover`/`-pressed`, `-secondary-hover`/`-secondary-pressed`, `-border`, `-outline-border`, `-tint-*` — the full set is in `button.css`), size → the ramp-multiplier pattern, and references `button.css`. A **structural** component reuses existing tokens — neutral `text-foreground`, the `text-label-*` / typography scale, `text-danger` (role colors resolve as `text-*` utilities, each registered as a `--color-*` in `src/tailwind/theme.css`), the shared `status-disabled` utility (`layers/utilities.css`) — and references `text.css` / `separator.css`, not `button.css`. Never hardcode a color; map spacing/size to its **semantic Tailwind alias** — `gap-small`, `p-medium`, `text-label-medium` (the `--fri-spacing-*` / `text-label-*` scales are mapped in `src/tailwind/theme.css`) — never a raw Tailwind numeric (`ms-0.5`, `gap-1`) nor the bare `gap-(--fri-spacing-*)` var form. States: which react-aria data attrs apply (`data-hovered`, `data-pressed`, `data-disabled`, `data-focus-visible`, plus component-specific).
5. **Plan the surfaces and demos against the generator templates.** The build scaffolds a fixed file set from `turbo/generators/templates/` — read the templates for the chosen kind. For `native`: `component.tsx.hbs`, `variants.ts.hbs` (emits `<name>.styles.ts`), `styles.css.hbs`, `stories.tsx.hbs`, `mdx.hbs`; for `aria`, the `.aria` variant of each (`component.aria.tsx.hbs`, `variants.aria.ts.hbs`, `styles.aria.css.hbs`, `stories.aria.tsx.hbs`, `mdx.aria.hbs`); plus the kind-agnostic `index.ts.hbs` (barrel) and `changeset.md.hbs`. **Compound** (a callable root plus subparts, like `link`'s `Link.Icon` or `grid`'s `Grid.Item`): also `namespace.ts.hbs` and one subpart stub per part — decide and record the subpart names and the shape (callable-root, scaffolded from the `parts` arg, versus `scroll-area`'s hand-wired namespace-of-parts object) in the issue so `component-implement` passes `parts` to `pnpm gen`. Plan the five surfaces and the showcase stories — the trio only ([`.claude/rules/stories-docs-sync.md`](../../rules/stories-docs-sync.md)): `Default`, plus `Variants` when the design has a `variant` axis (a color or shape axis joins that grid), plus `Sizes` when it has a `size` axis; states live in the doc's State selectors table, and use-case stories are the designer's, authored after ship — plan none. Then **follow the local pattern** (`.claude/rules/follow-local-pattern.md`): match the newest `bases/` neighbor on import order, naming, ordering, and prose terseness.
6. **Converge, then record the design to the issue.** Only after the design has CONVERGED (user approved in chat, no open decisions) and an **explicit, per-artifact go-ahead** to write to the shared tracker; until then the design lives in chat or a draft under the Claude project dir (`~/.claude/projects/<project>/blueprints/<name>.md`, outside the repo, never committed). Then record it to a GitHub issue:
   - **Preflight.** Confirm `gh auth status` succeeds and the repo has a GitHub remote (`git remote -v`). No GitHub remote → STOP and ask how the user tracks work.
   - **Create vs update: ASK, don't guess.** No relevant issue → confirm, then `gh issue create --title "<X>" --body-file ~/.claude/projects/<project>/blueprints/<name>.md`. A relevant issue the user named → `gh issue edit <n> --body-file …`. Never auto-match by title or topic; if unsure which, ask for the number. One yes authorizes one named artifact, never a batch.
   - **Use the repo's issue template.** Check `.github/ISSUE_TEMPLATE/` first; when one exists, read it and map the design onto its sections (YAML forms → emulate each field's label as a `### Label` heading). Pass the filled body via `--body-file` (never `--template`). Title per the repo's convention — discover it from commitlint / a title-validation workflow / `git log`, don't invent.
   - **Body carries:** the primitive + ladder + token/state map + surface/demo plan + any extra requirements the human gave, plus a **Global Constraints** block (the repo invariants — `fri-<name>` naming, the token ladder, `@layer theme/base/components/utilities` order, `src ↔ exports`, the contrast floor, every value distinct — copied verbatim from `CLAUDE.md` and `.claude/rules/architecture.md`). No placeholders: "the usual variants" or "like Button" → enumerate every value with its token.
7. **Write the implementation plan in the issue body**, after the design, at task altitude: each task names what it does, the files it touches, its **Done** (an observable behavior or contract) + **Verification** (the exact command). The tasks map onto the build contract — swap the primitive, fill the ladder, mirror the css, write the stories, the mdx — each a checkbox naming `component-implement` as the executor. The body then holds design + plan as one artifact; `component-implement` branches from the issue and ships it.

## Rules

- The spec — component, primitive, prop/variant/size/state set — comes from the human; never guess it, derive it from library docs alone, or pre-fill it. The library docs ground the answers; they do not replace the ask. An unspecified prop or variant is a question to ask open, never a default.
- Never recommend, suggest, label "recommended", or float an example menu for what to expose (props, variants, sizes, states); elicit it open and record it verbatim. Recommend only doc- or tree-grounded facts.
- Choose the primitive before the variants.
- The generator `primitive` kind is interactivity, not library: `aria` for interactive, `native` for display/layout including a non-interactive react-aria part (Label, Text).
- Reuse the repo vocabulary — `primary…danger`, `solid…plain`, `xs…xl`; never invent a color or size vocabulary when the repo set fits.
- Drop the color and visual-variant axes for a structural component (label, text, separator); use its domain axis plus boolean states. Every value distinct with its `fri-<name>-<value>` class; add a new axis only with a stated reason.
- Record the flat `tv({ base })` shape and `--fri-*` tokens the generator and build use; never a `slots: { root }` shape or a raw Tailwind numeric (`ms-0.5`, `gap-1`). Never hardcode a color; map every color to the `--fri-<role>` ladder.
- Design only from the live tree, the human's spec, and the library docs; never seed the shape from git history or a prior/dropped version.
- The issue body is the single source of truth — never a committed repo file; the draft (`~/.claude/projects/<project>/blueprints/<name>.md`) is the pre-ship `--body-file` buffer, outside the repo, never committed.
- No GitHub write without an explicit per-artifact yes, and only after the design has converged; a design approval is not a write authorization. One yes authorizes one named artifact, never a batch. Never auto-match an issue by title or topic — confirm the number with the user.
- Never ship an issue that still carries a `_(pending)_` decision — converge first. Take no code or build action.

## Acceptance criteria

- Re-read the ladder: no two values share a class or token; no color is hardcoded instead of a `--fri-<role>` slot; every value has its `fri-<name>-<value>` class. Fix any defect here.
- Primitive chosen and justified against the existing component it is most like; name, primitive, prop/variant spec, and extra requirements all came from the human, validated against the library docs — nothing guessed or derived from docs alone.
- Re-read the recorded body for placeholders (TBD/TODO gone), contradictions, and ambiguity before handing to `component-implement`; fix by editing the body (`gh issue edit <n> --body-file`).
- Sub-agent reviewer (**opus** — adversarial judgment): pressure-test the premise, ladder, and plan for gaps — a missing state, an ambiguous variant, an unprovable Done — before any build. Builder ≠ verifier.
